# ADR-001: Микросервисная архитектура

## Статус

Принято

## Контекст

Маркетплейс объединяет несколько доменов с разной нагрузкой и темпом изменений:

- каталог (много чтений, редкие записи продавцами);
- персонализация (CPU-bound ранжирование, эксперименты с алгоритмами);
- заказы и платежи (строгая консистентность, аудит);
- уведомления (асинхронная доставка, retry).

Монолит упростил бы начальную разработку, но создаёт единую точку деплоя и масштабирования.

## Решение

Разделить систему на **независимые микросервисы** по bounded context (User, Catalog, Feed, Order, Payment, Notification) с отдельными базами данных и независимым деплоем.

## Альтернативы

| Вариант | Плюсы | Минусы |
|---|---|---|
| Монолит | Простота, транзакции в одной БД | Сложно масштабировать каталог и feed отдельно |
| Модульный монолит | Компромисс на старте | Всё равно один деплой, риск «big ball of mud» |
| **Микросервисы** | Независимое масштабирование, изоляция сбоев | Сложность observability, распределённые транзакции |

## Последствия

- **Плюсы:** команды могут работать параллельно; Feed Service масштабируется отдельно при пиках трафика; сбой Notification Service не блокирует оформление заказа.
- **Минусы:** нужны API Gateway, service discovery, distributed tracing; saga-паттерн для сквозных транзакций (заказ + оплата + резерв).
- **Митигация:** начать с 6 сервисов, не дробить дальше без необходимости; использовать события для eventual consistency там, где допустимо.
